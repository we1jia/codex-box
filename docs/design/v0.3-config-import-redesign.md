# Codex Box · 配置命名空间与导入体验重设计 v0.3

> 状态：Draft  
> 日期：2026-06-27  
> 依据：ADR-0008、TasteSkill redesign protocol  

## 1. 设计结论

Codex Box 的配置体验要从“解释底层文件”改成“管理自己的配置中心，并兼容导入外部工具”。

核心改动：

- 主配置目录统一为 `~/.codex/codex-box/`。
- `~/.opencodex/` 降级为 OpenCodex 导入来源。
- `~/.opencodex/` 不参与运行时模型列表、API 服务链路或代理路由兜底。
- `8765` 降级为外部 OpenCodex 残留检测项。
- `1455` 是 Codex Box 默认本地连接端口，配置里只出现这一条 Codex Box 代理入口。
- Settings 增加“导入配置”入口，用于 OpenCodex、Codex++、CC Switch、Cockpit Tools 与 Codex 原生备份。

## 2. 当前对话与历史记录摘要

从线程 `019f046c-7206-7f01-8c76-63425e74bcc4` 与当前对话可以归纳出这些明确结论：

1. 用户只是要求参考 OpenCodex 的配置思路，不是要求 Codex Box 写入 OpenCodex 的 `8765` 端口。
2. Codex Box 自己的本地代理默认是 `1455`，生成配置时不应出现 `8765`。
3. `model_catalog_json` 必须是顶层配置，不能被拼接到 `[model_providers.*]` 表下面。
4. 会话归属 Provider 不能写死；新默认应是 `codex_model_router_v2 · Codex MultiRouter`，同时允许选择历史 Local，例如 `codex_local_access · Codex API Service`。
5. `~/.opencodex/` 不应该作为 Codex Box 主目录。Codex Box 应使用自己的目录，并支持从 OpenCodex 一键导入。
6. 官方订阅与 BYOK 代理不能混成一条认证链。官方订阅不抓 token，BYOK 走 Codex Box 本地代理。
7. 页面要更像“配置台”，减少 hash、路径、raw JSON、inject-map 等专家字段对普通用户的干扰。

## 3. 信息架构

### 3.1 Settings / Import

新增“导入配置”区块：

```text
导入配置
把其他 Codex 工具里的模型来源、模型目录和备份导入 Codex Box。

[扫描本机配置]

OpenCodex        发现 2 个来源、3 个模型      [预览导入]
Codex++          发现 CodexPlusPlus provider  [预览导入]
CC Switch        未发现，可手动选择目录        [选择目录]
Cockpit Tools    未扫描                         [选择 Data Directory]
Codex 备份       发现 5 个 config 快照          [查看]
```

### 3.2 Models / 模型配置

主流程不展示 `~/.opencodex`。路径文案改为：

```text
Codex Box 模型目录
~/.codex/codex-box/custom_model_catalog.json
```

当检测到 OpenCodex 旧目录：

```text
发现 OpenCodex 配置
可以导入到 Codex Box，不会修改原文件。

[预览导入]
```

### 3.3 Codex Runtime / 本地连接

状态卡只展示一条入口：

```text
本地连接
127.0.0.1:1455

Codex 配置：已指向 Codex Box
模型目录：Codex Box 模型目录
```

如果发现 `8765`：

```text
发现 OpenCodex 残留端口
当前配置指向 127.0.0.1:8765，这是 OpenCodex 的默认端口。Codex Box 不会启动这个端口。

[迁移到 Codex Box 连接]
```

API 服务链路表只展示 Codex Box 当前真实可路由的上游 API。旧 `opencodex` provider 或 `127.0.0.1:8765` 只能作为残留问题提示，不应作为“API 服务”行展示。

## 4. 交互流程

### 4.1 一键导入

```text
扫描
  -> 展示来源和数量
  -> 脱敏预览
  -> 用户勾选 provider/model/profile
  -> 写入 Codex Box 主目录
  -> 展示 config.toml diff
  -> 用户确认应用
```

导入预览必须显示：

- 来源工具
- 源文件路径
- 将写入的 Codex Box 文件
- provider 名称
- model id / display name
- 是否检测到明文 API Key
- 是否检测到 `8765`
- 将如何迁移到 `1455`

### 4.2 迁移旧 OpenCodex 文件

旧文件：

```text
~/.opencodex/providers.json
~/.opencodex/custom_model_catalog.json
```

目标文件：

```text
~/.codex/codex-box/providers.json
~/.codex/codex-box/custom_model_catalog.json
```

规则：

- 不删除旧文件。
- 保留未知字段。
- 明文 API Key 默认标记为“需要整理”，除非用户明确允许临时导入。
- 导入后 `model_catalog_json` 指向 Codex Box 主目录。

## 5. 视觉与文案原则

按 TasteSkill，本页面属于产品工具 redesign，不使用营销页式大 hero 或炫技动效。

- 主题：浅色 Mac Dashboard + frosted glass web approximation。
- 形状：卡片 `rounded-3xl`，按钮 `rounded-2xl`，输入 `rounded-2xl`。
- Accent：沿用现有蓝色/青色系统，不引入 AI 紫。
- 动效：只用于状态切换、展开详情、按钮反馈。
- 密度：保留开发工具的信息密度，但把专家字段折叠。

禁止：

- 同时展示 `1455` 和 `8765` 而不解释归属。
- 把 `~/.opencodex` 写成 Codex Box 主目录。
- 在 warning 里塞长路径导致横向滚动。
- 用 `Inject base_url`、`Proxy Runtime` 这类底层词当主按钮。

## 6. 验收标准

1. 新增模型后，默认写入 `~/.codex/codex-box/providers.json` 与 `~/.codex/codex-box/custom_model_catalog.json`。
2. “启用到 Codex”生成的 `model_catalog_json` 指向 `~/.codex/codex-box/custom_model_catalog.json`。
3. Codex Box 生成配置不出现 `127.0.0.1:8765`。
4. OpenCodex 旧目录可被扫描和导入，但不会被默认写入。
5. UI 文案使用“Codex Box 模型目录”“本地连接”“导入配置”，而不是把 OpenCodex 当主概念。
6. Runtime 的 API 服务链路不显示 `OpenCodex` / `opencodex`，除非是在“导入配置”或“旧配置残留”提示中。
7. 所有写入仍有 diff、backup、confirm 和 rollback。
