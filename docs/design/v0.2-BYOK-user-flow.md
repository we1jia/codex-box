# Codex Box · BYOK 用户配置体验设计 v0.2

> 目标：把 “providers.json / custom_model_catalog.json / config.toml / proxy inject” 包装成普通用户能理解和完成的配置流程。
> 状态：Draft · 2026-06-25
> 适用范围：Overview、Models、Provider Routes、Codex Runtime、Profiles、Diagnostics、Settings

---

## 1. 设计结论

当前页面把底层实现直接暴露给用户，导致用户需要先理解 `api_key`、`${ENV_VAR}`、provider、catalog、runtime、inject-map 才能使用。新的体验应该改成：

> 先让用户完成“添加模型来源 → 选择模型 → 启用到 Codex”三步，专家细节折叠到“高级详情”里。

参考 `/Users/liuweijia/Desktop/AI/opencodex` 运行中的 `http://localhost:8765/dashboard`，真正值得吸收的不是深色 UI 或源码实现，而是它的产品组织方式：

> 一个 “Gateway Config” 页面同时完成新增模型、保存 API Key、更新 Codex 下拉框、重启 Codex、还原原生配置。

Codex Box 应保持自有 Mac Dashboard 视觉和安全写入闭环，但主流程应向这个“一页式模型配置台”靠拢。

核心原则：

- 默认给用户一条最短路径，不让用户手写 JSON。
- 页面主文案使用用户语言：模型来源、API Key、可选模型、启用到 Codex。
- 文件路径、hash、diff、raw schema、环境变量名属于高级信息，默认收起。
- 出错时说“发生了什么 + 怎么修”，不说“parse failed / api_key_ref invalid”。
- 安全红线不放松：真实 API Key 不显示、不写日志、不保存到配置文件。

## 2. 用户心智模型

用户只需要理解 4 个概念：

| 用户看到的概念 | 含义 | 底层对应 |
|---|---|---|
| 模型来源 | API 服务商，比如 MiniMax、DeepSeek、OpenRouter | `~/.opencodex/providers.json` |
| 可选模型 | Codex App 下拉里要出现的模型，比如 MiniMax-M3 | `~/.opencodex/custom_model_catalog.json` |
| 启用到 Codex | 让 Codex App 通过本机代理看到这些模型 | `proxy start` + `inject config.toml` |
| 当前方案 | 当前使用哪组模型、来源和运行配置 | `~/.codex/config.toml` profile |

用户不应该在主流程里看到：

- `content_hash`
- `providersContentHash`
- `catalogContentHash`
- `inject-map.json`
- raw JSON
- `api_key_ref`
- `wire_api`
- `base_url` 的技术细节（除非展开高级设置）

## 3. 主流程

### 3.1 首次使用

首屏应该显示一个“模型配置台”，而不是把用户丢到模型目录、Provider 路由、Runtime 三个页面之间：

```
┌───────────────────────────────┬──────────────────────────────┐
│ 新增模型配置                   │ Codex 下拉框模型              │
│                               │                              │
│ 模型名称                       │ ☑ MiniMax M3                 │
│ 接口地址                       │ ☑ DeepSeek Chat              │
│ API Key                        │ ☐ GLM-4                      │
│ □ 保存后提醒重启 Codex Desktop │                              │
│                               │ □ 更新后提醒重启 Codex Desktop│
│ [保存并添加该模型]             │ [更新下拉框菜单]              │
└───────────────────────────────┴──────────────────────────────┘
```

用户仍然完成三件事，但页面上不是拆成三页：

1. 添加模型来源：选择服务商、填写 API Key、保存为安全引用。
2. 添加可选模型：从推荐模型中选择，或手动输入模型 ID。
3. 启用到 Codex：启动本地代理并注入 Codex 配置。

### 3.2 已有配置但有风险

如果检测到明文 Key：

```
需要处理 2 个模型来源
Codex Box 发现有 API Key 直接写在配置文件里。为了安全，已暂时不使用这些 Key。

受影响：custom_i5jf、custom_okw6

[一键整理为安全配置]  [稍后处理]
```

点击“一键整理为安全配置”进入迁移向导：

1. 展示将保留哪个来源、合并哪些重复来源。
2. 让用户确认环境变量名称，例如 `MINIMAX_API_KEY`。
3. 把真实 Key 写入系统环境变量或提示用户复制命令。
4. 写回 `${MINIMAX_API_KEY}` 引用。
5. 生成备份和可回滚记录。

### 3.3 空模型目录

如果 Provider 存在但模型目录为空：

```
还没有可选模型
你已经添加了模型来源，但还没有告诉 Codex 哪些模型要显示在下拉里。

[添加推荐模型]  [手动添加模型]
```

推荐模型应根据 provider 自动给出：

| Provider | 推荐模型 |
|---|---|
| MiniMax | `MiniMax-M3` |
| DeepSeek | `deepseek-chat`、`deepseek-reasoner` |
| OpenRouter | 让用户手动选择或填模型 ID |
| OpenAI Official API | `gpt-5.1`、`gpt-5-mini` |

### 4.3 Provider Routes

定位：管理“API 服务商和 Key”。

页面标题建议改名为“模型来源”，并从主流程降级为高级页。普通用户不应该先进入这里。

页面结构：

```
模型来源
说明：模型来源是 API 服务商，例如 MiniMax、DeepSeek、OpenRouter。

[添加来源]

┌ 来源列表 ───────────────────────────────┐
│ MiniMax        已连接       1 个模型      │
│ DeepSeek       需要 API Key  0 个模型      │
└────────────────────────────────────────┘

┌ 来源详情 ───────────────────────────────┐
│ 服务商：MiniMax                         │
│ API 地址：api.minimax.io                │
│ API Key：已安全保存为 MINIMAX_API_KEY    │
│ 可选模型：MiniMax-M3                    │
│ [测试连接] [编辑] [删除]                 │
└────────────────────────────────────────┘
```

明文 Key 风险态：

```
需要修复 API Key
这个来源的 Key 直接写在配置文件里。Codex Box 已暂时不使用它。

[整理为安全配置]
```

高级详情折叠：

- `base_url`
- `wire_api`
- `${ENV_VAR}`
- `http_headers`
- raw provider JSON

### 4.4 Codex Runtime

定位：控制“Codex App 是否能看到这些模型”，并承载运行期排障能力。

用户语言：

- “本地连接”替代 “proxy runtime”
- “启用到 Codex”替代 “Inject base_url”
- “恢复原配置”替代 “Restore base_url”

页面结构：

```
Codex 运行时
状态：未启用 / 运行中 / 需要处理

本地连接：127.0.0.1:1455
可用模型：1 个
Codex 配置：已启用

[启动连接] [启用到 Codex] [恢复原配置]
```

这页从主流程降级为“连接状态 / 高级操作”。普通用户在 Models 页点“保存并添加该模型”或“更新下拉框菜单”时，系统应提示下一步需要启动本地连接、启用到 Codex；失败时再引导到 Codex 运行时页排查。

操作解释必须靠近按钮：

- 启动连接：只启动 Codex Box 本机服务，不修改 Codex 配置。
- 启用到 Codex：会修改 `~/.codex/config.toml`，写入前显示 diff 并备份。
- 恢复原配置：从备份映射还原原来的 provider 地址。
- 运行日志：只展示脱敏后的本地 runtime 事件，真实 API Key 永远不进入日志。
- 会话管理：属于运行时上下文，不属于模型配置；入口放在 Codex Runtime 页。

### 4.5 Profiles

定位：管理“当前工作方案”，不是让用户理解 TOML profile。

页面标题建议：

```
工作方案
说明：一个工作方案包含模型、权限、网络和 MCP 设置。
```

详情展示顺序：

1. 当前模型
2. 模型来源
3. 权限模式
4. 网络
5. MCP
6. 高级配置

### 4.6 Diagnostics

定位：回答“为什么不能用”。

诊断结果按用户动作分组：

| 分组 | 检查项 |
|---|---|
 模型来源 | API Key 是否安全、API 地址是否可达 |
 可选模型 | 模型目录是否为空、模型是否绑定到来源 |
 Codex 连接 | 本地连接是否运行、Codex 配置是否指向本机 |
 安全写入 | 备份目录、diff、atomic write |

每个问题必须给一个按钮：

- “修复 API Key”
- “添加模型”
- “启动连接”
- “启用到 Codex”

## 5. 推荐的最简单配置

以 MiniMax 为例，用户表单只需要填：

| 字段 | 用户看到 |
|---|---|
 服务商 | MiniMax |
 API Key | 用户粘贴真实 Key |
 默认模型 | MiniMax M3 |

保存后底层生成：

`~/.opencodex/providers.json`

```json
{
  "providers": [
    {
      "name": "minimax",
      "base_url": "https://api.minimax.io/v1",
      "wire_api": "chat",
      "api_key": "${MINIMAX_API_KEY}",
      "enabled": true
    }
  ]
}
```

`~/.opencodex/custom_model_catalog.json`

```json
{
  "models": [
    {
      "slug": "MiniMax-M3",
      "model": "MiniMax-M3",
      "display_name": "MiniMax M3",
      "provider": "minimax",
      "backend_model": "MiniMax-M3",
      "backend_provider": "minimax",
      "visibility": "list"
    }
  ]
}
```

真实 Key 存储：

```bash
MINIMAX_API_KEY=...
```

UI 不应该让用户看见这三份内容，除非点开“高级详情”。

## 6. 交互细节

### 6.1 添加模型来源

表单：

```
选择服务商
[ MiniMax v ]

API Key
[ sk-•••••••••••••••••••• ]

默认模型
[ MiniMax M3 v ]

[测试并保存]
```

保存行为：

1. 先用 Key 测试 `/v1/models` 或 provider health。
2. 测试成功后把 Key 写入系统环境变量。
3. 配置文件只写 `${MINIMAX_API_KEY}`。
4. 自动创建默认模型条目。
5. 显示“下一步：启用到 Codex”。

### 6.2 修复明文 Key

用户不需要理解“明文 Key”。

推荐文案：

```
为了安全，需要整理 API Key
这个 Key 现在直接保存在配置文件里。Codex Box 可以帮你把它移到系统环境变量，并把配置文件改成安全引用。

将执行：
1. 备份当前配置
2. 保存 Key 到 MINIMAX_API_KEY
3. 把配置文件中的 Key 替换为 ${MINIMAX_API_KEY}

[整理]
```

### 6.3 启用到 Codex

推荐文案：

```
让 Codex App 看到这些模型
Codex Box 会启动一个只监听本机的连接，并把 Codex 配置指向它。

写入前会展示变更并自动备份。

[预览变更] [确认启用]
```

## 7. 实施优先级

### P0：立刻解决困惑

1. Models 改名为“模型配置”，成为主入口。
2. Models 顶部加入“新增模型配置”表单：模型名称、接口地址、API Key、保存后提醒重启。
3. Models 右侧加入“Codex 下拉框模型”：checkbox、Vision Bridge 标记、删除、更新下拉框菜单。
4. Provider Routes 改名为“模型来源”，从主流程降级为高级页。
5. Codex Runtime 保持“Codex 运行时”命名，从主流程降级为状态/排障页。
6. 顶部 warning 改成用户语言，不出现长路径和 hash。
7. Overview 增加“继续设置”主入口。

### P1：最小可用向导

1. 新增后端命令 `simple_model_config_save`：一次写入 provider + catalog，并保存 API Key 为环境变量引用。
2. 内置 MiniMax、DeepSeek、OpenRouter、OpenAI API 预设。
3. 保存来源时自动创建默认模型。
4. 保存成功后自动启动本地连接并提示是否启用到 Codex。
5. 增加“一键整理为安全配置”。

### P2：完整闭环

1. 添加模型选择器，支持从 provider `/v1/models` 拉取。
2. 支持 Vision Bridge 配置，但默认放到折叠区。
3. Diagnostics 每个问题提供修复按钮。
4. Settings 中增加“高级模式”开关，默认隐藏底层文件字段。
5. 会话管理放入 Codex Runtime 的高级区；语音助手不进入 BYOK 主线，后续作为独立扩展页评估。

## 10. OpenCodex Dashboard 对照

实际运行 `http://localhost:8765/dashboard` 后，OpenCodex 页面结构如下：

| OpenCodex 区块 | 用户价值 | Codex Box 写法 |
|---|---|---|
| 顶部状态：Active & Intercepting | 告诉用户是否正在接管 Codex | Overview / Models 顶部显示“Codex 下拉已启用 / 未启用” |
| 重启 Codex | 保存后立刻生效 | Models 表单里保留“保存后提醒重启 Codex Desktop”；自动重启作为后续独立能力评估 |
| 还原原生 | 用户能撤销 | Codex 连接页保留“恢复原配置”，Models 页保存后给撤销入口 |
| Gateway Config tab | 核心主流程 | 合并进 Codex Box 的 Models/模型配置页 |
| API Settings & Keys | 新增模型配置 | 左侧表单：模型名称、接口地址、API Key |
| Model Dropdown Customizer | 控制下拉框 | 右侧列表：checkbox、删除、更新下拉框 |
| Vision Fallback | 视觉降级能力 | 先作为折叠高级配置，不作为 P0 |
| macOS Permissions | Computer Use 权限 | 与 BYOK 主线弱相关，放 Diagnostics 或后续扩展 |
| Voice Assistant tab | 语音助手 | 不进入 BYOK 主线 |
| Session Manager tab | 会话管理 | 放入 Codex Runtime 的高级区，不进入 Models 主流程 |
| Live Console Logger | 排障日志 | Codex 连接 / Diagnostics 的高级日志区 |

重要差异：

- OpenCodex 会把 API Key 写进 `providers.json`；Codex Box 不能照搬这一点。
- Codex Box 的保存动作应做到“用户粘贴 Key，但文件只保存 `${ENV_VAR}` 引用”。
- Codex Box 仍然必须 backup → diff → confirm → atomic write → rollback。

## 8. 文案规范

| 不要写 | 改成 |
|---|---|
 `api_key_ref invalid` | API Key 需要整理 |
 `providers.json parse failed` | 模型来源配置需要处理 |
 `custom_model_catalog.json 为空` | 还没有可选模型 |
 `Inject base_url` | 启用到 Codex |
 `Proxy Runtime` | Codex 本地连接 |
 `content_hash` | 高级详情：内容指纹 |
 `wire_api` | 高级详情：接口格式 |

## 9. 成功标准

一个新用户应该能在 3 分钟内完成：

1. 选择 MiniMax。
2. 粘贴 API Key。
3. 看到 `MiniMax M3` 出现在模型列表。
4. 点击“启用到 Codex”。
5. 在 Codex App 模型下拉里看到模型。

过程中不需要手动打开 JSON 文件，不需要知道 `${ENV_VAR}` 是什么，不需要理解本地代理。
