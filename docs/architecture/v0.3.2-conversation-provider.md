# Codex Box · 会话归属 Provider 与本地代理路由

> 状态：v0.3.2 草案 · 2026-06-26  
> 目标：让用户可编辑 `model_provider` 的显示/归属名称，同时保证实际请求稳定走 Codex Box 本地代理。

---

## 1. 结论

Codex Box 把 `model_provider` 分成两层理解：

1. **会话归属 Provider**：写入 `~/.codex/config.toml` 的 `model_provider`，用于 Codex App 识别当前会话归属、历史会话列表和 provider 分组。
2. **真实上游 Provider**：由 `~/.opencodex/custom_model_catalog.json` 中模型条目的 `backend_provider` 指向，再由 `~/.opencodex/providers.json` 找到真实 `base_url` / `wire_api` / `api_key_ref`。

因此用户可以把会话归属 Provider 设置为历史上用过的 `openai`、`custom`、`codex_local_access`、`minimax` 等名称，用来找回旧会话列表；实际请求仍统一进入 `http://127.0.0.1:<port>/v1`，再由本地代理按模型 id 路由到真实上游。

---

## 2. 为什么不能写死

Codex Box 是给多用户使用的工具，不能假设某个固定 provider 名称。

每个用户的历史配置可能来自：

- Codex 官方默认：`model_provider = "openai"`
- AITabby/opencodex 或其他工具：`model_provider = "custom"`
- 用户自定义：`model_provider = "codex_local_access"`
- 某个模型来源名：`model_provider = "minimax"` / `deepseek` / `moonshot`

所以 UI 必须从当前用户本机配置扫描候选项，而不是写死候选值。

---

## 3. 扫描来源

后端命令：`conversation_provider_candidates`

扫描范围：

| 来源 | 路径 | 用途 |
|---|---|---|
| 当前配置 | `~/.codex/config.toml` | 当前生效的 `model_provider` 与 `[model_providers.*]` |
| 旧备份 | `~/.codex/config.toml.bak*` | 发现迁移前的 provider 名称 |
| 独立配置 | `~/.codex/*.config.toml` | 兼容用户手动保存的 profile/config 文件 |
| Codex Box 备份 | `~/.codex/codex-box/backups/*.toml` | 发现 Codex Box 历史写入前状态 |

扫描结果按来源和时间排序，当前配置优先。若没有显式 `model_provider`，默认候选和当前值为 `openai`。

## 4. 写入规则

后端命令：

- `conversation_provider_preview`：生成 diff，不写文件。
- `conversation_provider_apply`：用户确认后写入。

写入必须遵守项目写入闭环：

```text
读取当前 config
  -> 校验 expected_hash
  -> 生成 diff
  -> 用户确认
  -> backup ~/.codex/config.toml
  -> atomic write
  -> 失败 rollback 到 backup
```

### 4.1 `provider_id = "openai"`

`openai` 是 Codex 内置 provider，不能写成 `[model_providers.openai]` 自定义表。目标配置：

```toml
model_provider = "openai"
openai_base_url = "http://127.0.0.1:1455/v1"
model_catalog_json = "/Users/<user>/.opencodex/custom_model_catalog.json"
```

含义：Codex App 仍认为 provider 是 OpenAI；但请求地址被导向 Codex Box 本地代理。

### 4.2 自定义 provider_id

例如 `custom` / `codex_local_access` / `minimax`。目标配置：

```toml
model_provider = "codex_local_access"
model_catalog_json = "/Users/<user>/.opencodex/custom_model_catalog.json"

[model_providers.codex_local_access]
name = "Codex API Service"
base_url = "http://127.0.0.1:1455/v1"
wire_api = "responses"
requires_openai_auth = true
experimental_bearer_token = "dummy"
request_max_retries = 3
stream_max_retries = 3
stream_idle_timeout_ms = 600000
```

含义：Codex App 仍按该 provider 名称归属会话；但 `[model_providers.<id>].base_url` 指向本地代理。

## 5. 前端表单

位置：`Codex Box Runtime` 页面。

表单字段：

| 字段 | 说明 |
|---|---|
| 历史 Provider | 从本机配置和备份扫描得到的候选项 |
| `Provider ID` | 可编辑，写入 `model_provider` |
| 显示名称 | 写入 `[model_providers.<id>].name`，`openai` 模式下仅用于 UI 回显 |
| `wire_api` | 默认 `responses`，可选 `chat` |
| 端口 | 生成 `http://127.0.0.1:<port>/v1` |
| 历史 `base_url` | 只用于回显来源，不作为真实请求地址写回 |
| `requires_openai_auth` | 自定义 provider 默认保持 `true`，用于兼容 Codex Desktop 登录态检查 |

用户点击“预览变更”后必须看到 diff，确认后才写入。

---

## 6. 请求路径保证

不论会话归属 Provider 叫什么，Codex App 的请求都应进入：

```text
http://127.0.0.1:<port>/v1
```

真实上游选择由模型目录决定：

```text
Codex App selected model id
  -> Codex Box local proxy
  -> custom_model_catalog.json 找 model/backend_model/backend_provider
  -> providers.json 找 backend_provider 对应 base_url/env_key/wire_api
  -> 转发到真实上游
```

关键约束：不要把可编辑的 `model_provider` 当成真实路由 provider；真实路由必须使用 `backend_provider`。

---

## 7. 安全边界

- 不抓 token。
- 不 patch Codex Desktop。
- 不接管系统代理。
- 不把 API Key 写入 `config.toml`、日志或诊断报告。
- 所有 `~/.codex/config.toml` 写入必须 backup、diff、confirm、atomic write、rollback。
