# Codex Box · 架构 v0.3.1 · BYOK 本地代理 runtime

> 目标：让 Codex App 原生 picker 真正看到并切换 BYOK 模型,切换后请求真实路由到对应 provider。
> 路线：在 Codex Box Rust 后端内置一个 127.0.0.1 的本地 HTTP 代理,完全独立实现(不 spawn 外部进程,不复用 AITabby/opencodex 源码/UI)。
> 对应：[PRD.md](../../PRD.md) §1-§10 · [v0.3-BYOK 架构](./v0.3-BYOK.md) · [v0.3-BYOK 数据模型](../data-model/v0.3-BYOK.md) · [ADR-0006](../decisions/0006-本地代理-runtime.md)

---

## 1. 背景与决策回顾

v0.3 的"BYOK 模型下拉"主线([ADR-0005](../decisions/0005-BYOK-模型下拉与多-provider-路由.md))已建立配置侧闭环:

- 安全读写 `~/.codex/config.toml` 与 `~/.codex/codex-box/{providers.json, custom_model_catalog.json}`
- 写入走 backup → diff → confirm → atomic write → rollback
- secret 走 env 引用
- UI 控制台 + 前端页面就绪

但 v0.3 的"BYOK"实际只解决了"配置文件可以改"的问题。**没有解决两个真正的关键问题**:

1. **Codex App picker 真的能看到这些模型吗?** — 答案是不一定。Codex App 自己的 picker 不会做跨 `[model_providers.*]` 段的模型合并;它只对当前 active profile 的 `model_provider.base_url` 发请求,具体展示哪些 model 取决于 Codex App 内部实现。
2. **切换后请求真的路由到对应 provider 吗?** — 答案取决于用户手填的 `base_url` 是否可直接打。如果用户给 `https://api.deepseek.com/v1` 这样的 OpenAI-compatible URL,Codex App 自己的请求适配层大概率能 work(因为 Codex App 内部就是按 OpenAI Responses/Chat 协议写的),但**统一性、鉴权注入、模型命名空间、隐藏/可见性切换、合并展示**等能力 Codex App 不提供。

**真正的 BYOK 体验(Cursor 那样,在同一个下拉里看并切换所有来源的模型)**需要 Codex App 看到一个**统一的、fake 的"一个 provider"**。这就是本地代理 runtime 的作用。

## 2. 代理 runtime 模式

### 2.1 整体链路

```
                ┌───────────────────────────────────────────────────────────┐
                │                                                           │
                ▼                                                           │
   Codex App  /  /v1/models  ───┐                                            │
        │      /v1/chat/completions ──┐                                       │
        │      /v1/responses     ──┐   │                                       │
        │                          │   │                                       │
        │       ┌──────────────────▼───▼────────────────────┐                  │
        │       │ Codex Box 本地代理 (127.0.0.1:{port})      │                  │
        │       │   /v1/models: 合并 [model_providers.*]    │                  │
        │       │             + custom_model_catalog         │                  │
        │       │   /v1/chat, /v1/responses: 按 model id     │                  │
        │       │              fan-out + 注入鉴权/headers    │                  │
        │       └─┬───────────┬───────────┬──────────────────┘                  │
        │         │           │           │                                     │
        │         ▼           ▼           ▼                                     │
        │   openai.com   openrouter   智谱/DeepSeek/Moonshot/...               │
        │   (官方 API)   (第三方)     (国产 OpenAI-compatible)                  │
        │                                                                       │
        │   官方订阅/原生 GPT: 优先走 openai-official managed route,无 route │
        │   时才走 native fallback;只透传本次请求上下文,不读本地 token       │
        │                                                                       │
        └───────────────────────────────────────────────────────────────────────┘
```

**关键点**:
- Codex App 始终只看到**一个** `model_provider`,即 `codex-box`(base_url=`http://127.0.0.1:{port}/v1`;默认 `1455`,端口冲突或手动指定时以 MultiRouter 预览中的端口为准)
- 它发给 Codex Box 的 `model` 字段优先是 `custom_model_catalog.json` 中的裸 `slug/model_id`(例:`glm-4`、`minimax`、`gpt-5`);代理按 catalog 的 `backend_provider/backend_model` 解析真实上游。`provider/model` 仅作为手动调试和冲突兜底格式保留。
- 官方订阅/原生 GPT 不走第三方 BYOK translator:catalog entry 使用 `provider="openai"` 且不写 `backend_provider` 时,代理优先匹配 router provider 里的 `openai-official` route(`auth.source = "managed_codex_oauth"`),并把请求转向 `https://chatgpt.com/backend-api/codex`;没有该 route 时才进入 native OpenAI fallback。
- Codex Box **不**读取本地官方账号 token 文件,也不把官方登录态转换成用户可见 API key。

### 2.2 与 AITabby/opencodex 的差异

| 维度 | AITabby/opencodex v0.2.2 | Codex Box |
|---|---|---|
| 入口进程 | `dist/server.js`(Node) | Codex Box Tauri 进程内 tokio task |
| 代理端口 | 127.0.0.1:8765 | 127.0.0.1:1455(可改) |
| 启动 | 外部 PM2 / launchd | Tauri Builder setup 阶段自动拉起 |
| 关闭 | 外部 kill | Tauri exit hook 优雅 shutdown |
| UI | 自带 Web Dashboard | 集成到 Codex Box Mac Dashboard + frosted glass |
| License | 仓库自带 | 项目自有(非 AGPL) |
| 实现语言 | TypeScript | Rust + axum + reqwest |
| Codex Desktop 集成 | `autoPatchCodexConfig()` | `proxy_inject_base_url_*` Tauri command(同写入闭环) |
| 模型目录 | `~/.opencodex/custom_model_catalog.json` | `~/.codex/codex-box/custom_model_catalog.json` |
| Provider 路由 | `~/.opencodex/providers.json` | `~/.codex/codex-box/providers.json` |
| Codex Box 注入元数据 | 内部文件 | `~/.codex/codex-box/inject-map.json` |
| 鉴权 | api_key 内联 | `${ENV_VAR}` 引用(不落盘) |
| 协议转换 | Anthropic↔OpenAI responses/chat 互转 | v1 先按 `wire_api` 分流:`responses` provider 优先 passthrough 到 `/responses` 并做最小 normalize;`chat` provider 才走 Codex Desktop `ws://.../v1/responses` → OpenAI-compatible `/chat/completions` 的桥接;已补 reasoning/tool-call/history 核心状态机 |
| 周边功能 | Voice Companion / Computer Use / Vision Bridge | **不**实现(与 BYOK 无关) |
| 自定义 dashboard 路由 | `/dashboard` (SSE 日志) | 集成到 Codex Box 桌面端 `/v1/models` |

**Codex Box 的核心差异**:
1. **不依赖外部 runtime**:Codex Box 关了 = 代理关,不需要 PM2 / launchd
2. **不复制代码/UI**:所有代理代码独立实现,Rust + axum;UI 走 Codex Box 自己的 frosted glass 风格
3. **安全闭环更强**:API key 强制 env 引用,绝不内联到任何文件
4. **写入闭环统一**:`config.toml` / `inject-map.json` / `providers.json` / `catalog.json` 全走同一套 backup→diff→confirm→atomic→rollback
5. **OpenCodex 只做兼容导入**:`~/.opencodex/*.json` 可扫描和导入,但不作为 Codex Box 主写入目录,也不作为实时模型列表或代理路由兜底

## 4. 路由解析

model id 解析规则(`src-tauri/src/proxy/routing.rs`):

1. **原生 OpenAI catalog**:`provider="openai"` 且没有 `backend_provider` 的 visible entry 用于 GPT 原生订阅/官方能力在同一 picker 中展示。请求解析时优先使用 MultiRouter 自动生成的 `openai-official` managed route;只有没有 route 时才进入 native OpenAI fallback。
2. **custom catalog**:优先按 `custom_model_catalog.json` 的 `model_id` 查 entry,再按 `backend_provider/backend_model` 定位真实上游。
3. **MultiRouter provider**:若 `backend_provider` 指向的 `providers.json` 条目带 `codexRouting`,则按 `routes[].match.models`、`routes[].match.prefixes`、`defaultRouteId` 选择 route,再转到 `targetProviderId` 对应的真实 provider;`upstream.modelMap` 负责 picker model id → 上游真实 model id 改写,`upstream.apiFormat` 映射为 `wire_api`。
4. **命名空间形式** `provider/model`:精确查表,model id 必须在该 provider 的 models 列表里(列表为空时透传),主要用于手动调试。
5. **裸 model id** `model`:在 inject-map provider models 里查,唯一命中时返回。
6. **冲突**返回 404(`model_not_found`)。

旧 OpenCodex 残留 provider(名称为 `opencodex` 或 base_url 指向 `127.0.0.1:8765` / `localhost:8765`)不会进入 `/v1/models`,也不会被 MultiRouter 当作真实上游。若 catalog 命中 router provider 但 `codexRouting` 没有解析到合法真实上游,代理直接返回未命中,不能回退到 router 自身或旧 8765 端口。

例:

```
请求 model="minimax"        → 先查 catalog,按 backend_provider/backend_model 路由
请求 model="minimax-m3"     → catalog 命中 codex_model_router_v2,codexRouting 再转到 minimax/MiniMax-M3
请求 model="zhipu/glm-4"    → 手动命名空间格式,匹配 zhipu provider
请求 model="glm-4"          → inject-map 唯一命中 zhipu 时转发
请求 model="glm-4" (有 zhipu + openai 都提供) → 404,要求 catalog 明确 backend_provider 或手动使用命名空间形式
```

## 5. 端点详细

### 5.1 `GET /v1/models`

- 数据源:inject-map providers + `~/.codex/codex-box/custom_model_catalog.json`(`visible=true`)
- 输出:OpenAI 标准 schema
- 加响应头 `X-Codex-Box-Source: merged`(方便 debug)
- **跳过**官方订阅 provider 对象(`codex-subscription`)本身;原生 GPT 模型通过 catalog 中 `provider="openai"` 且无 `backend_provider` 的条目展示
- 模型 id 优先使用 catalog 里的裸 `slug/model_id`,对齐 AITabby/opencodex 的 picker 行为;inject-map 里的 provider models 也返回裸 model id,不为空 provider 生成占位模型
- catalog 条目会同时投影 snake_case 与 camelCase 元数据,包括 `displayName`、`contextWindow`、`maxContextWindow`、`defaultReasoningEffort`、`supportedReasoningEfforts`、`inputModalities`、`supportsParallelToolCalls`、`backendProvider/backendModel` 等,兼容 Codex Desktop 不同 picker 读取路径

### 5.2 `POST /v1/chat/completions`

- 解析 body 拿 `model`,按路由解析 provider
- `upstream_url = provider.original_base_url + "/chat/completions"`
- 注入:`Authorization: Bearer <${ENV_VAR}>` + provider.http_headers
- SSE 透传(reqwest `bytes_stream` → axum `Body::from_stream`)
- 透传 status code / response headers(去除 hop-by-hop)

### 5.3 `POST /v1/responses`

- `wire_api=responses` provider 直接转发到真实上游 `/responses`,并对 Responses SSE 做最小 normalize
- `wire_api=chat` provider 会走同一套 Responses→Chat translator;若请求 `stream=true`,上游也保持 Chat Completions streaming,代理逐行把 Chat SSE 转回 Responses SSE,不再等待完整 JSON 后伪造流式响应
- Chat fallback streaming 同步维护 conversation history,并把末尾 usage chunk 映射回 Responses `usage`

### 5.4 `GET /v1/responses` WebSocket upgrade

Codex Desktop 在部分会话里会把 Responses 调用发成：

```text
ws://127.0.0.1:{port}/v1/responses
```

因此代理必须同时支持 HTTP `POST /v1/responses` 和 WebSocket upgrade。

当前实现：

- 接收 WebSocket `response.create` 消息。
- 从消息中读取 `model`，按 inject-map / custom catalog 解析 `backend_provider` 与 `backend_model`。
- 若真实 provider 的 `wire_api = "responses"`：
  - 将请求体 `model` 改写为 `backend_model`
  - 优先转发到真实上游 `{provider.base_url}/responses`
  - 上游 Responses SSE event 直接转发回 Codex Desktop WebSocket
  - 只做最小 normalize：过滤 `<think>...</think>` 污染、保留原生 Responses 事件语义
- 若真实 provider 的 `wire_api = "chat"`，才将 Responses 请求转换为 OpenAI-compatible `/chat/completions` 请求：
  - `instructions` → system message
  - `input[].content[].input_text` → chat message content
  - `input[].content[].input_image` / `image` → 非文本-only 上游转为 OpenAI-compatible chat `image_url` content block;文本-only 上游转为明确占位或走 vision bridge
  - `input[].content[].input_file` / `input_audio` → Chat Completions `file` / `input_audio` content block;纯文本模型用明确占位替代
  - `input[].function_call` → assistant `tool_calls`
  - `input[].function_call_output` → tool message + `tool_call_id`
  - `input[].custom_tool_call` / `tool_search_call` / 对应 output → Chat `tool_calls` / tool message,用于多轮 history 恢复
  - `tools` / `tool_choice` → Chat Completions `tools` / `tool_choice`,覆盖 function、custom、namespace、tool_search;custom tool 会把原始工具定义以稳定 JSON 嵌入 description,保留 `format` / grammar 等元数据;若工具为空则丢弃 `tool_choice` 与 `parallel_tool_calls`,避免兼容上游返回 400/503
  - `tool_search_output.tools` → 补充本轮可用 Chat tools,避免动态加载工具后 `tool_choice` 被错误丢弃
  - `max_output_tokens` → `max_tokens`
  - Codex reasoning 请求 → Chat 上游推理开关:优先读取 route/provider/catalog 的 `codexChatReasoning` 声明,支持 `thinkingParam`(`thinking` / `enable_thinking` / `reasoning_split`)与 `effortParam`(`reasoning_effort` / `reasoning.effort`);未声明时再按模型/平台兜底推断,MiniMax 改写为 `reasoning_split`,Qwen/DashScope/Bailian/SiliconFlow 改写为 `enable_thinking`,Kimi/Moonshot/GLM/Zhipu 改写为 `thinking = { type }`
  - 流式 Chat fallback 自动补 `stream_options.include_usage = true`,并把 Chat `usage` 映射回 Responses `usage.output_tokens_details.reasoning_tokens`
  - `model` 改写为 `backend_model`
- 已知纯文本模型的图片块会被替换为 `[Unsupported Image]`,避免上游把图片结构误解释成普通文本或工具标签
- 请求真实上游 `{provider.base_url}/chat/completions`;WebSocket 与 HTTP streaming 都保持真实上游流式,不会把 `stream=true` 降级成非流式请求。
- 将上游 chat SSE chunk 转换回 Responses WebSocket events：
  - `response.created`
  - `response.output_item.added` / `response.content_part.added` for reasoning
  - `response.reasoning_text.delta` / `response.reasoning_summary_text.delta`
  - `response.reasoning_text.done` / `response.reasoning_summary_text.done`
  - `response.output_item.added`
  - `response.content_part.added`
  - `response.output_text.delta`
  - `response.output_text.done`
  - `response.content_part.done`
  - `response.function_call_arguments.delta`
  - `response.function_call_arguments.done`
  - `response.output_item.done`
  - `response.completed`
- 非流式 Chat JSON 返回与 HTTP JSON fallback 复用同一个 synthetic delta converter,保留 `reasoning_content` / `reasoning`、`tool_calls`、`usage` 和 final `finish_reason`,避免 MiniMax 等 OpenAI-compatible 上游在 WebSocket 路径丢 reasoning/tool call。
- Chat fallback 会按 `client_metadata.session_id` 或请求头 `x-session-id/session-id/x-thread-id/thread-id` 维护内存 conversation history；`previous_response_id` 存在时合并新消息并用 overlap 去重，避免 Codex Desktop 未发送完整历史时丢上下文。
- MultiRouter 路由解析会把 catalog/provider/`codexRouting.routes[].capabilities` 里的 `textOnly`、`inputModalities` 传给 translator。若 route 或 catalog 明确声明文本-only,`input_image` 不再直接透传为 Chat `image_url`,而是降级为可读占位,避免 MiniMax-M3、DeepSeek V4 这类纯文本上游收到图片块后返回 400 或生成错误工具调用。
- 基础 vision bridge 已接入 Chat fallback 与 Responses passthrough:当 catalog 或 route 启用 `vision_bridge_enabled` / `visionBridge` 并配置 `vision_fallback_base_url`、`vision_fallback_model`、`vision_fallback_api_key_ref` 时,代理会先调用 OpenAI-compatible 视觉模型,把用户消息里的 `input_image` 替换为 `[截图描述: ...]` 文本,再交给 MiniMax 等文本上游。该实现复用 env key,不落盘 secret;OpenCodex 的图片压缩、持久描述缓存可作为后续增强。

边界：

- 不读取、不转发、不复用 ChatGPT 官方账号 token。
- native OpenAI forward 若只收到 `Authorization: Bearer PROXY_MANAGED`,会在本地返回 `native_openai_auth_unresolved`,不会把占位符发给 `chatgpt.com` 或 `api.openai.com`。
- 官方订阅只透传 Codex Desktop 当前请求可见的官方认证上下文;Codex Box 当前不实现独立的 Codex OAuth token 管理器。
- 当前已覆盖 BYOK/OpenAI-compatible chat streaming 的核心路径、`wire_api=responses` 的 Responses passthrough/minimal normalize 路径，以及 Chat fallback 下的 reasoning/tool-call/conversation-history/基础 vision bridge 状态机。OpenCodex 的 computer-use 参数补丁、语音/伴侣周边能力不属于 Codex Box 当前主线。

### 5.5 `GET /healthz`

- 内存状态:`{ status, port, uptime_ms, provider_count }`
- 不走网络,不暴露 key

### 5.5 错误体

OpenAI 风格:

```json
{ "error": { "message": "...", "type": "model_not_found", "code": "model_not_found" } }
```

status code 透传到 Codex App(404/400/401/502 等)

## 6. 注入 / 还原写入闭环

**Inject(启动前或启动后都可)**:把 `~/.codex/config.toml` 的 `[model_providers.*].base_url`(除 `codex-subscription` 和旧 `opencodex/8765` 残留外)改写为 `http://127.0.0.1:{port}/v1`,原值记入 `inject-map.json`。

Tauri command:
- `proxy_inject_base_url_preview({port})` → 返回 diff + 新的 inject-map + 校验用 hash,**不**写文件
- `proxy_inject_base_url_apply({preview, confirmed: true})` → 写 config.toml + 写 inject-map.json,走 backup→atomic→rollback
- UI:Codex Box Runtime 页"Inject base_urls → proxy"卡片

**Restore(停止代理时)**:从 inject-map 反向写回原 base_url,清空 inject-map。

Tauri command:
- `proxy_restore_base_url_preview()` → 返回 diff
- `proxy_restore_base_url_apply({preview, confirmed: true})` → 写 config.toml + 清空 inject-map.json
- UI:Codex Box Runtime 页"Restore base_urls"卡片

**MultiRouter Sync(Models 页 + Runtime 主操作)**:把当前 `providers.json` + `custom_model_catalog.json` 收敛成一个稳定的 `codex_model_router_v2` 本地 router provider,并让 `config.toml` 指向 Codex Box 自有模型目录。该操作负责让 Codex App picker 在同一个下拉里看到官方原生 GPT 条目和多家 BYOK/API 模型;第三方模型进入本地 router 后按 `codexRouting.routes` 分流,原生 GPT 条目自动生成 `openai-official` managed route。旧 `codex_local_access` 仍作为兼容历史桶识别和归并来源,但不再作为新 MultiRouter 默认写入目标。

Tauri command:
- `codex_multirouter_preview({proxyPort, routerProviderId, ensureCodexConfig})` → 返回 `providers.json`、`custom_model_catalog.json`、`config.toml`、`models_cache.json`、`inject-map.json` 五份 diff + 校验 hash,**不**写文件
- `codex_multirouter_apply({preview, confirmed: true})` → 写 router provider、为官方 GPT 生成 `openai-official` route、重写第三方 catalog backend provider、更新 `model_catalog_json`,并把合并后的官方 GPT + BYOK/API 模型目录同步到 Codex App 热缓存 `~/.codex/models_cache.json`;若 `inject-map.json` 只剩或包含旧 `opencodex/8765` 残留,也会按预览 diff 清理。每份文件走 backup→atomic→rollback
- 若当前 `~/.codex/models_cache.json` 没有可导入的官方 GPT/Codex 原生模型,MultiRouter 预览会用 Codex Box 内置的官方默认候选补齐 native OpenAI catalog entries。这样 GPT 默认模型会进入真实 `custom_model_catalog.json` / `models_cache.json` diff,不只是 Models 页面前端只读展示。官方请求仍只透传 Codex Desktop 当前请求自带的登录上下文,Codex Box 不读取、不保存官方 token。
- MultiRouter 生成第三方 route 时必须写入可解析的 `capabilities`:至少包含 `apiFormat` 与 `inputModalities`;MiniMax、DeepSeek V4/Chat/Reasoner、Qwen3-Coder、GLM/Kat/LongCat 等已知文本模型默认标记 `textOnly = true`。MiniMax route/catalog 还必须声明 `codexChatReasoning.thinkingParam = "reasoning_split"`,让 Codex 的 reasoning 请求在 Chat fallback 中转成 MiniMax 可识别参数。模型名带 `vision` / `multimodal` / `omni` / `image` / `-vl` / `_vl` 等原生视觉提示时不套用文本-only 默认。
- `codex_models_cache_restore_preview()` → 只针对 Codex Box 标记过所有权的 `models_cache.json` 生成恢复/删除 diff,**不**写文件
- `codex_models_cache_restore_apply({preview, confirmed: true})` → 若存在 `models_cache.codex-box-backup.json` 则恢复原始官方缓存,否则删除 Codex Box owned cache,让 Codex App 下次自行重建
- `codex_desktop_integration_status()` → 只读检测 `config.toml`、`models_cache.json`、`auth.json` 结构字段和 Codex Desktop 进程/CDP 状态,用于解释“catalog/cache 都正确但下拉框仍被 renderer 白名单过滤”的场景
- `codex_desktop_picker_unlock()` → 用户显式点击的实验动作;仅当 Codex Desktop 已经带 `--remote-debugging-port` 运行时,通过 CDP 向当前 renderer 注入下拉框白名单补丁,补 `available_models/use_hidden_models`、`Response.prototype.json`、`model/list` 消息路径、`app-server-manager-signals-*` 的 `list-models-for-host` 返回值,并修正 React auth context 为 `chatgpt` 以保留官方订阅模型。不自动启动/重启 Codex Desktop,不写配置文件,不修改 `app.asar` 或 renderer 文件。
- UI:Models 页"Codex MultiRouter"卡片和 Runtime 页主按钮共用同一条 preview/apply 链路;Runtime 页会先确保本地代理可用,再生成五份写入预览,用户确认 diff 后才同步
- `proxy_route_test({modelId, includeImage, performUpstream})` → Runtime 页的路由测试。`performUpstream=false` 是默认 dry-run,不请求真实上游,只用当前 catalog/providers/inject-map 模拟 Codex `/v1/responses` 请求,验证 resolver 是否命中、`wire_api=chat` 时 Responses→Chat translator 是否可生成请求、text-only 模型是否会错误向上游发送 `image_url`。`performUpstream=true` 是用户显式点击的真实上游测试:关闭流式、限制输出 token、注入 env auth/header 后请求真实 `/chat/completions` 或 `/responses`,只返回 status code/latency,不展示响应正文或密钥。`managed_codex_oauth` 官方订阅 route 会跳过真实上游测试,因为它需要 Codex Desktop 当前请求里的 ChatGPT 登录上下文。

写入 `config.toml` 时必须同时投影:
- 顶层 `model_catalog_json` 与 `model_provider = "<routerProviderId>"`;默认 router id 为 `codex_model_router_v2`
- `[model_providers.<routerProviderId>]` 的 `base_url`、`wire_api = "responses"`、`requires_openai_auth = true`、`experimental_bearer_token = "PROXY_MANAGED"`;其中 `base_url` 必须使用本次 MultiRouter preview/apply 的 `proxyBaseUrl`,不能固定写死默认端口
- `supports_websockets = false`,避免 Codex Desktop 把 router provider 当成内置 OpenAI WebSocket 直连语义
- provider 内联 `models = [...]`,从 `custom_model_catalog.json` 的 visible 条目生成,用于覆盖部分 Codex Desktop custom picker 只读 provider 内模型数组的路径

写入 `models_cache.json` 时必须:
- 复用现有 `client_version`,避免 Codex App 拒绝非当前客户端版本的缓存
- 写入 `etag = "codex-box-model-catalog"` 标记所有权
- 只在已有 cache 能读取到 `client_version` 时写;缺失时跳过 cache 同步,仍保留 `config.toml` 与 catalog 投影
- 首次覆盖非 Codex Box owned cache 前,先把原始内容原子写入同目录 `models_cache.codex-box-backup.json`;该文件只作为恢复官方缓存的 sidecar,不进入常规模型导入流程
- 用户点"恢复官方缓存"时,只有当前 `models_cache.json` 仍带 `etag = "codex-box-model-catalog"` 才允许恢复;存在 sidecar 则原子恢复 sidecar 并删除 sidecar,不存在 sidecar 则删除 owned cache
- `opencodex_config_read()` 从 native cache 导入官方 GPT 模型时跳过该 owned cache,避免把 Codex Box 合并目录里的 BYOK 模型误导入为 `openai`
- 普通读取页面只把官方 native cache 合并到内存视图,不写 `custom_model_catalog.json`;只有用户确认 `codex_multirouter_apply` 后,官方 GPT 条目才会随 MultiRouter 同步一起写入 Codex Box catalog 与 Codex App cache。若 native cache 缺失,预览/apply 使用内置默认 GPT/Codex 候选补齐真实 catalog entries。

**Diagnostics(只读)**:Diagnostics 页会同时读取 `effective_routing_status` 与 `codex_desktop_integration_status`,分组展示:

- `config.toml` 是否能解析,避免重复 `model_catalog_json` 这类 TOML 错误被吞掉
- 当前 `model_provider` / `model` / `model_catalog_json` 与请求入口 `base_url`
- router provider 是否保持 `requires_openai_auth = true`,避免 Codex Desktop 隐藏 ChatGPT 登录态和官方能力
- `providers.json` 是否存在 `openai-official` managed route,并检查 `auth.source = "managed_codex_oauth"` 与 `baseUrl = "https://chatgpt.com/backend-api/codex"` 是否匹配
- `models_cache.json` 是否由 Codex Box 接管、是否保留 `client_version`、模型数量
- `auth.json` 是否存在 ChatGPT 登录态结构;只读结构字段,不读取或展示 token
- Codex Desktop 是否已经普通启动且没有 `--remote-debugging-port`;若模型数大于官方少量默认集合,提示 renderer Statsig/白名单过滤风险,并提供显式下拉框解锁动作

默认诊断仍然只读。下拉框解锁必须由用户显式点击触发,且只对已经开放 CDP 的 Codex Desktop renderer 生效;如果 Codex Desktop 是普通启动,命令只返回 `needs_remote_debugging`,不会杀进程或自动重启。

**关键不变量**:
- 每次写都先 backup 到 `~/.codex/codex-box/backups/{ts}-{hash}.{toml,json}`
- diff 必须先在 UI 展示,用户点 confirm 才写
- 写失败自动 rollback 到最近一次 backup

## 7. 进程生命周期

```
Tauri Builder.setup
   └─ manage(Arc<ProxyState>::new())
   └─ (auto_start 默认 true,后续接 Settings 开关)

Codex Box Runtime 页:
   点 Start
      └─ proxy_start({ port: 1455 })
         └─ probe_port (1455..1460)
         └─ TcpListener::bind("127.0.0.1", port)
         └─ tokio::spawn(axum::serve(...))
         └─ state.set_status(Running)
         └─ 写 runtime-state.json
   点 Stop
      └─ proxy_stop()
         └─ state.signal_shutdown()  →  serve task 收到信号后退出 accept 循环
         └─ state.set_status(Stopped)
         └─ 写 runtime-state.json
   点 Restart
      └─ proxy_restart({ port })
         └─ stop + sleep 500ms + start

Tauri 进程退出
   └─ on_window_event 触发 Drop → 代理随进程自动结束
```

**安全**:
- 永远只 bind `127.0.0.1`
- 不绑 LAN,不绑 IPv6,不绑 Unix socket
- 不接外部网络代理(`HTTPS_PROXY` 等不被读取)
- 错误体不包含上游响应 body,只记 status / latency / provider name

## 10. 风险与边界

| 风险 | 缓解 |
|---|---|
| Codex App 版本升级改变 picker 行为 | Codex Box 对 Codex App 呈现的就是 base_url=127.0.0.1:{port} 的本地 provider,大多数 picker 协议变化不影响 |
| 国产模型 API 协议差异(非标准 OpenAI-compatible) | provider.http_headers 暴露自定义 header;`/chat` 还是 `/responses` 可在 provider.wire_api 选;v1 范围透传 body 不做协议转换 |
| MiniMax 等 provider 声称支持 Responses 但事件细节与 Codex Desktop 不完全一致 | `wire_api=responses` 优先 passthrough 到 `/responses`,做 `<think>` 等最小 normalize;若 route/catalog 显式启用 vision bridge,Responses passthrough 也会先把 `input_image` 改写为截图描述文本。Chat fallback 已补 reasoning/tool/history/image_url/input_file/input_audio/custom tool metadata/usage 状态机,并支持 route/provider/catalog 的 `codexChatReasoning` 显式声明,把 Codex reasoning 转成 `reasoning_split` / `enable_thinking` / `thinking` / `reasoning.effort` |
| SSE 大流量性能 | reqwest + axum body stream 在 macOS 千 token/s 场景下表现良好;10K+ token/s 未压测;v2 再优化 |
| env 引用缺失 | 代理启动时**不**做检查;请求时 env 缺失则日志记 warn,代理返回 401(由 upstream 返回);用户在 Diagnostics 看到"env var not set"warn |
| 注入后 Codex App 旧模型仍存在 | Codex App 需重启或重读 config 才能看到新模型;UI 提示用户"重启 Codex App" |
| catalog/cache 已合并但 Codex Desktop 下拉仍只显示官方少量模型 | Diagnostics 检测 Codex Desktop 是否普通启动且无 CDP,并明确提示 renderer 白名单过滤风险;若 Codex Desktop 已带 CDP 启动,用户可显式点击下拉框解锁注入 renderer 补丁 |
| 端口冲突 | start 前 probe_port,占用则 +1,最多 5 次;都失败返回 AppError |

## 11. 不做什么(v1 范围)

- ❌ 不做 OpenCodex 的 voice / computer-use 专项补丁
- ❌ 不做模型 router(按 task 选 model);当前用户手动切 model
- ❌ 不做请求日志持久化(只有 tracing 日志,脱敏)
- ❌ 不做 systemd / launchd 安装(Codex Box 关了代理就停)
- ❌ 不接外部代理服务器(`HTTPS_PROXY` / `ALL_PROXY` 不读)
- ❌ 不做 fallback 链(主 provider 失败时换备用)
- ❌ 不做 rate limit / token 计数(留给上游)
- ❌ 不做 voice / computer use(与 BYOK 无关)
