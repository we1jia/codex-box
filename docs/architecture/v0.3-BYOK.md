## 1. 一句话架构

Codex Box 是一个 Tauri 桌面控制台,前端负责 Mac Dashboard 体验,Rust 后端负责 `~/.codex/config.toml` 与 `~/.opencodex/*.json` 的安全读写、Codex Desktop 安装检测、provider URL 健康检查,完全不依赖外部 gateway 进程。

```
Codex Box (Tauri)
  ├─ React UI
  │  ├─ Overview
  │  ├─ Models              (模型下拉预览 + 切换)
  │  ├─ Provider Routes     (多 provider 路由管理)
  │  ├─ Codex Box Runtime   (127.0.0.1 代理控制台:start/stop/inject/restore)
  │  ├─ Profiles            (工作配置)
  │  ├─ Diagnostics         (脱敏诊断)
  │  └─ Settings            (应用设置)
  ├─ Rust Backend
  │  ├─ commands/           (Tauri command 边界)
  │  ├─ config/             (TOML 读写、backup、diff、atomic write)
  │  ├─ opencodex_config/   (~/.opencodex/*.json 读写)
  │  ├─ codex_runtime/      (Codex Desktop / CLI / CODEX_HOME 检测)
  │  ├─ health/             (config / runtime / provider / env 诊断)
  │  └─ proxy/              (本地代理 runtime:127.0.0.1:1455,/v1/models + 路由转发)
  ├─ ~/.codex/config.toml              (Codex App 读 base_url = 127.0.0.1:1455 后,所有请求经代理)
  ├─ ~/.codex/codex-box/
  │  ├─ config.json                    (Codex Box 自有设置)
  │  ├─ backups/                       (backup 记录)
  │  ├─ inject-map.json                (代理注入的原 base_url 备份)
  │  └─ runtime-state.json             (代理当前状态,供 UI 刷新可见)
  ├─ ~/.opencodex/
  │  ├─ providers.json
  │  └─ custom_model_catalog.json
  └─ Codex Desktop / Codex CLI         (只读检测;运行时代理请求路径)
```

详细代理实现见 [`v0.3.1-BYOK-proxy.md`](./v0.3.1-BYOK-proxy.md)。

## 2. 分层职责

| 层 | 职责 | 不做什么 |
|---|---|---|
| React UI | 页面、导航、表单、diff 确认、状态展示 | 不直接读写用户文件 |
| Tauri commands | 前后端命令边界、参数校验、错误脱敏 | 不承载复杂业务状态 |
| `config/` | `~/.codex/config.toml` 读取、解析、diff、backup、atomic write、rollback | 不做 UI 决策 |
| `opencodex_config/` | `~/.opencodex/*.json` 读取、写入、未知字段保留 | 不解析或落盘明文 secret |
| `codex_runtime/` | Codex Desktop / CLI / CODEX_HOME 检测 | 不修改 Codex Desktop 内部文件 |
| `health/` | config / runtime / provider / env 诊断 | 不输出 secret 明文 |
| `secret/` | env / Keychain 引用和脱敏 | 不持久化 token 明文 |
| `proxy/` | 127.0.0.1 本地 HTTP 代理 runtime(端口 1455),合并 `/v1/models`、按 model id 转发 `/v1/chat/completions` 与 `/v1/responses` | 不绑 LAN、不读系统代理 env、不做协议转换(v1)、不抓 token |

## 3. 页面结构

主导航收敛为 8 个页面:

| 页面 | 主要能力 |
|---|---|
| Overview | 全局状态、活跃 profile、模型目录摘要、快速入口 |
| Models | 模型下拉预览(订阅 / 官方 / 第三方 / 本地)+ 切换 |
| Provider Routes | provider 增删改启停、可见性 toggle |
| Codex Box Runtime | **本地代理控制台**:状态 / start / stop / restart / inject base_url / restore base_url / 路由表 / /v1/models 预览(v0.3.1 新增) |
| Profiles | 工作配置管理 |
| Diagnostics | 脱敏诊断报告 |
| Settings | 应用设置、日志、备份、实验开关 |

`Custom Model Catalog` 作为 `Provider Routes` 页面的子区块(用户加模型 = 加 catalog 条目 + 加 provider 路由)。稳定后再拆独立页面。

`Config Diff` 作为相关页面中的能力区块,稳定后再拆成独立页面。

## 4. 核心数据流

### 4.1 Dashboard 读取

```
Overview 挂载
  -> invoke("dashboard_summary")
  -> 读取 config_snapshot
  -> 读取 opencodex_providers
  -> 读取 opencodex_model_catalog
  -> 读取 codex_runtime_status
  -> 聚合 health 摘要
  -> 渲染状态卡
```

### 4.2 BYOK 配置写入(模型切换)

```
用户发起"切换到 智谱 GLM-4"
  -> preview_config_change({
       operation: "UpdateProfile",
       profile: <existing profile with model_provider="zhipu", model="glm-4">
     })
  -> 校验 profile、provider、env_key
  -> 生成 diff + warnings + current_hash
  -> 用户确认
  -> apply_config_change(expected_hash)
  -> backup ~/.codex/config.toml -> ~/.codex/codex-box/backups/{ts}-{hash}.toml
  -> atomic write (tmp -> rename)
  -> 失败 rollback
  -> 同时按需更新 ~/.opencodex/providers.json(AITabby 约定)
  -> 返回新 snapshot
```

### 4.3 新增 Provider

```
用户填入 provider 表单(name, kind, base_url, wire_api, env_key, models)
  -> preview_provider_add
  -> 在 ~/.codex/config.toml 的 [model_providers.<name>] 段预览 diff
  -> 在 ~/.opencodex/providers.json 预览 diff
  -> 用户确认
  -> apply_provider_add
  -> backup 两个文件
  -> atomic write 两个文件
  -> 失败 rollback 两个文件
  -> 返回新 snapshot
```

### 4.4 Codex Runtime 检测(只读)

```
Codex Runtime 页挂载
  -> invoke("codex_runtime_probe")
  -> 探测 Codex Desktop.app 是否安装(LaunchServices / Spotlight)
  -> 探测 Codex CLI 是否在 PATH 中
  -> 探测 CODEX_HOME 路径与可读性
  -> 返回 CodexRuntimeStatus
  -> UI 展示:installed / not-installed / partial
  -> 不修改任何文件
```

## 5. 写入闭环(共用,适用所有 .toml 与 .json)

```
用户发起变更
  -> preview_config_change(request)
     ├─ 读取当前文件 -> 解析为结构化对象
     ├─ 应用变更(内存中)
     ├─ 序列化回原文(保格式)
     ├─ 生成结构化 diff + 文本 diff
     ├─ 计算 current_hash
     └─ 返回 ConfigChangePreview

  -> 用户在 UI 确认
  -> apply_config_change(expected_hash)
     ├─ 校验 current_hash == expected_hash
     ├─ 创建 backup(file_path, reason)
     ├─ 写 .tmp
     ├─ rename .tmp -> 原文件
     ├─ 成功:返回新 snapshot + backup_id
     └─ 失败:rollback 到 backup,返回错误

  -> 后台记录 BackupRecord(供回滚)
```

## 6. 与 AITabby/opencodex 的协作边界

Codex Box **不** spawn 外部 AITabby/opencodex 进程,仅复用其文件路径与字段约定:

- 共享 `~/.opencodex/providers.json` 路径与字段语义
- 共享 `~/.opencodex/custom_model_catalog.json` 路径与字段语义
- 写入时"未知字段保留",避免与 AITabby CLI 互相覆盖
- 读 AITabby CLI 写入的 providers.json,字段不识别则原样保留

详细事实见 [docs/references/aitabby-opencodex.md](../references/aitabby-opencodex.md)。

## 7. 安全边界

- 写入强制 backup → diff → confirm → atomic write → rollback
- 本地代理**仅**监听 127.0.0.1(端口默认 1455,可改),**不**绑 LAN / IPv6 / Unix socket
- **不**接管系统全局代理;代理内部**不**读 `HTTPS_PROXY` / `ALL_PROXY` 等环境变量
- 不抓取、不复用、不转发官方账号 token
- 不绕过官方登录
- 不规避 rate limit / 账号配额
- **不**默认修改 Codex Desktop 内部文件
- 日志不记录 prompt、文件正文、API key、token、password
- secret 在 config 中只允许 `${ENV_VAR}` 引用,不允许 inline value
- 代理请求时**按需懒读** env,**不**缓存明文 key,**不**把 key 写回日志或诊断报告

## 8. 模块落地顺序

1. `opencodex_config/{providers,catalog}.rs`:读取/写入 `~/.opencodex/*.json`,保格式,未知字段保留
2. `config/parser.rs` 扩展:解析 `~/.codex/config.toml` 时输出 `ModelProvider` 列表(已就绪,只补 model 字段)
3. `commands/model_catalog.rs`:Tauri command 边界,提供 `model_catalog_*` 系列命令
4. `commands/provider_route.rs`:Tauri command 边界,提供 `provider_route_*` 系列命令
5. `codex_runtime/locator.rs`:Codex Desktop / CLI / CODEX_HOME 检测(已就绪)
6. `health/provider.rs`:provider URL 可达性、env 变量存在性
7. 前端 Models / Provider Routes / Codex Runtime 三个 page 重写
8. 跑 `pnpm build` 验证前端构建通过
