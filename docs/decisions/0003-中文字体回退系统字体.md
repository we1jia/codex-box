# ADR 0003 · 中文字体回退到系统字体

> 状态:Accepted(反转 ADR-0001 的中文字体部分)
> 日期:2026-06-25
> 决策者:项目组

---

## 背景

ADR-0001 引入 4 个 fontsource 包(Inter / Noto Sans SC / Noto Serif SC / JetBrains Mono)
作为主字体,核心收益是跨平台一致 + 衬线大标题字重有效(Songti SC 没有 500/600)。

经过 M0 ~ M1 阶段实际使用,重新评估后,中文字体部分(Noto Sans SC + Noto Serif SC)被判定为
**投入产出比偏低**:

1. **bundle 体积**:Noto Sans SC 400 + 500 ≈ 2.2MB,Noto Serif SC 600 ≈ 1.5MB,合计约 3.7MB 原始
   / ~2.5MB gzipped。对一个面向本地 Codex 配置的轻量工具来说,这个占比偏大,Tauri 安装包
   显著膨胀。
2. **首屏等待**:fontsource 默认 `font-display: swap`,中文段落首屏会先用系统字体渲染、
   webfont 加载完成后再切换,**视觉上有一次"跳字"**,尤其是衬线大标题从 Songti SC 跳到
   Noto Serif SC 600 时,字重变化明显,反而不如一直用系统字体稳定。
3. **跨设备一致性回归到「系统差异」可以接受**:Codex Box 是 macOS-first 桌面应用,目标用户
   高度集中在 macOS 平台。Windows / Linux 是次要平台,系统字体差异(SimSun vs 苹方 vs Noto
   CJK)在该用户群中不是核心痛点。

英文 / 等宽部分(Inter Variable + JetBrains Mono)继续保留 fontsource 加载,这两个包加起来
约 0.5MB,价值明显(避免 macOS 系统 UI 字体替代带来的视觉割裂 + 数字等宽),不参与本次反转。

## 决策

**中文字体从 webfont(fontsource)回退到系统字体**,衬线 / 无衬线两个角色继续保留,
由系统字体承担:

| 角色 | 字体来源 | macOS | Windows | Linux |
|---|---|---|---|---|
| 中文衬线(品牌大标题) | 系统 | Songti SC / STSong | SimSun / FangSong | Noto Serif CJK SC |
| 中文无衬线(正文 / UI) | 系统 | PingFang SC / Hiragino Sans GB | Microsoft YaHei | Noto Sans CJK SC |
| 英文 / 数字 / 等宽 | webfont(fontsource) | — | — | — |

具体动作:

- 卸载 `@fontsource/noto-sans-sc`、`@fontsource/noto-serif-sc` 两个包
- `src/styles/fonts.css` 移除 Noto SC 的 `@import`,仅保留 Inter Variable + JetBrains Mono
- `tailwind.config.js` 的 `font-serif` / `font-sans` stack 去掉 Noto SC,补充跨平台系统中文回退
- `src/index.css` 的 `body` font-family 同步,去掉 Noto Sans SC

英文 / 等宽的 OpenType 特性(`tnum 1, ss01 1, calt 1, liga 1, cv11 0`)和
中英文字距 / 行高 token 全部保留,数字对齐、字距、行高行为不变。

## 权衡(必须接受的)

### 衬线大标题 `font-semibold` 不再生效

macOS Songti SC / STSong 只有 Regular 一档字重,Windows SimSun 同理。
Dashboard 的 `<h1 className="font-serif ... font-semibold">` 在反转后,
`font-semibold` 在中文部分会回退到 Regular,与英文 / 数字字符的 600 字重**不再视觉一致**。

**缓解**:
- 中文字符在 32px 大字号下,字重差异视觉影响小于 14-18px
- 大标题混排的英文(如 "Codex Box")仍然走 Inter Variable,可拿到 600 字重
- 后续如果用户反馈衬线大标题"分量不足",可走 ADR 单独评估

### 跨设备中文表现差异

- macOS:PingFang SC / Songti SC,SF Pro 系,现代
- Windows:Microsoft YaHei / SimSun,偏传统
- Linux:Noto Sans CJK / Noto Serif CJK,接近 Noto 设计但来自系统包

接受这个差异,与 ADR-0001 的"跨平台一致"目标相反,但**与"轻量 + 跨设备差异可接受"的
新优先级一致**。

## 备选

### A · 保留 Noto Sans SC,只去 Noto Serif SC(部分反转)

把衬线大标题也改成无衬线,只保留正文 webfont。**否决**:衬线品牌感是 v0.1 design spec
的明确选择,贸然去掉会破坏调性。

### B · 完全回退到系统字体(本次采纳,英文也去掉)

进一步把 Inter Variable / JetBrains Mono 也去掉。**否决**:英文 / 等宽 webfont 体积小
(0.5MB),收益大(视觉割裂 + 数字对齐),不参与反转。

### C · 引入子集化(只打包项目用到的中文字符)

用 `cn-font-split` / `fonttools` 二次子集化,目标 ~500KB。**否决**:需要构建链改动
(fontsource → fonttools 流水线),M0 时间盒不划算,留作 M5 之后再评估。

## 影响

### 正面

- **bundle 减重约 3.7MB 原始 / ~2.5MB gzipped**——Tauri 安装包显著瘦身
- **首屏无中文 webfont 等待**,去掉"跳字"问题
- **`pnpm install` 速度提升**,node_modules 减小
- **依赖简化**——少两个 fontsource 包,版本维护成本降低

### 负面 / 成本

- **跨设备中文表现差异**——见权衡部分
- **衬线大标题字重限制**——见权衡部分
- **font-semibold 在衬线大标题上效果减弱**——已知,接受

## 验证

1. `pnpm dev` 启动 Vite 开发服务器
2. DevTools Network 面板确认**不再有** Noto Sans / Noto Serif SC 的 woff2 加载请求
3. 视觉验证项:
   - 衬线大标题「早上好,开发者」使用系统 Songti SC(SimSun),字重看起来比之前细
   - 「Codex Box」英文保持 Inter Variable 600
   - 「当前 Codex 配置一切正常」中文使用系统 PingFang SC(YaHei),无字体切换
   - 「⌘1」「⌘2」数字 / 英文保持 Inter,等宽对齐
4. `pnpm build` 通过,产物大小对比本次修改前减少 ~2.5MB gzipped
